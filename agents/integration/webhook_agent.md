## WEBHOOK INTEGRATION PROTOCOL

### PHASE 1 — SECURITY & SIGNATURE VALIDATION
Webhooks are exposed public endpoints. You must verify identity before processing:
1. **HMAC Verification**: Retrieve the signature from the header (e.g., `Stripe-Signature` or `X-Hub-Signature`).
2. **Secret Key**: Retrieve the signing secret from the environment.
3. **Payload Hashing**: Re-compute the HMAC hash (SHA256) of the raw request body using the secret key.
4. **Constant-Time Comparison**: Use secure string comparison (preventing timing attacks) to match calculated signature vs. header signature.

---

### PHASE 2 — IDEMPOTENCY & DEDUPLICATION (Data Integrity)
Webhooks are guaranteed to deliver "at least once," which means we will receive duplicates:
- **Idempotency Key**: Identify the unique ID in the payload (e.g., `evt_123` or `event_id`).
- **Database Check**: Query a transaction log or Redis cache to verify if the event ID has already been successfully processed.
- **Locking**: Implement database transactions or distributed locks (Redis Redlock) to prevent race conditions from concurrent duplicate webhooks.

---

### PHASE 3 — ASYNCHRONOUS INGESTION QUEUE (Scaling)
Never run long-running business logic in the HTTP thread that receives the webhook.
- **Fast Ack**: Read signature → Write raw payload to queue (RabbitMQ, AWS SQS, or Celery) → Return `HTTP 200 OK` immediately (usually within < 100ms).
- **Background Worker**: Worker picks up the event from the queue and processes it asynchronously.

---

### PHASE 4 — ERROR RETRY & DEAD LETTER QUEUE (DLQ)
When database updates or business logic fail during processing:
- **Retry Count**: Retry the event up to 5 times.
- **Dead Letter Queue**: If retries are exhausted, move the payload to a DLQ for manual inspection.
- **Alerting**: Trigger alerts for consecutive DLQ failures.

---

## WEBHOOK INTEGRATION ANTI-PATTERNS (Never Do These)

```
✗ Never process webhooks synchronously (external calls during webhook response can block threads and cause timeouts).
✗ Never process payloads without verifying the HMAC signature (allows attackers to spoof events and inject fake data).
✗ Never write non-idempotent operations (could lead to charging users twice or double crediting accounts).
✗ Never use simple string comparison (`==`) for signature validation; always use constant-time comparison.
✗ Never return HTTP 500 error responses to third parties if the bug is our internal logic (triggers unwanted retries from their server; queue it instead).
```

## 2. Ingestion & Security Verification Blueprint (Python Code)
```python
import hmac
import hashlib
from fastapi import Request, HTTPException, status
from os import environ

async def verify_github_signature(request: Request):
    signature_header = request.headers.get("X-Hub-Signature-256")
    if not signature_header:
        raise HTTPException(status_code=400, detail="Missing signature")
        
    secret = environ.get("GITHUB_WEBHOOK_SECRET").encode("utf-8")
    body = await request.body()
    
    # Compute HMAC SHA256
    hash_object = hmac.new(secret, body, hashlib.sha256)
    expected_signature = "sha256=" + hash_object.hexdigest()
    
    # Constant-time comparison
    if not hmac.compare_digest(expected_signature, signature_header):
        raise HTTPException(
            status_code=status.HTTP_403_FORBIDDEN, 
            detail="Invalid signature"
        )
```
